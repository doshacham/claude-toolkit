# Explanation Methodology

Follow this methodology to produce clear, memorable, and thorough explanations of any concept, whether for a glossary entry, a teaching session, or inline context.

## Explanation Template

Include these sections in every explanation, in order:

### 1. One-Line Definition

Open with a single, precise sentence that captures the essence. Avoid jargon in this line — aim for clarity a beginner could grasp.

**Pattern:** "[Term] is [what it is] that [what it does / why it matters]."

### 2. Analogy or Metaphor

Provide a real-world analogy that maps onto the concept's core mechanism. Choose analogies from everyday life — physical objects, common experiences, familiar systems.

**Pattern:** "Think of it like [familiar thing] — [how the mapping works]."

Keep the analogy to 1-3 sentences. Do not over-stretch it.

### 3. Core Explanation

Explain the concept in technical detail. Cover:

- **What it is** — the mechanism, structure, or principle
- **How it works** — the process, flow, or behavior
- **Why it exists** — the problem it solves or the need it addresses

Use concrete language. Prefer short paragraphs (2-4 sentences each). Break complex ideas into numbered steps when a process is involved.

### 4. Practical Example

Provide one concrete, minimal example that demonstrates the concept in action. For programming concepts, include a short code snippet. For non-programming concepts, describe a real scenario.

Keep examples small — just enough to illustrate the point. Annotate key parts.

### 5. Key Takeaways

List 2-4 bullet points summarizing the most important things to remember. These should be independently useful — someone reading only this section should get the gist.

### 6. Related Concepts (Optional)

When relevant, list 2-3 related terms or concepts that the reader might want to explore next. Use the format: `**Term** — one-line description`.

## Writing Guidelines

- Write in plain, direct language
- Lead with the "what" before the "why"
- Use active voice and present tense
- Define jargon before using it
- One idea per paragraph
- Prefer concrete over abstract

## Quality Checklist

Before finalizing any explanation, verify:

- [ ] One-line definition is jargon-free and self-contained
- [ ] Analogy maps accurately to the concept
- [ ] Core explanation covers what, how, and why
- [ ] Example is minimal and annotated
- [ ] Key takeaways are independently useful
- [ ] Language is plain and direct
