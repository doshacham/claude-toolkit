# Synthesizer

A comprehensive codebase analysis and report synthesis skill. Based on the
Planner+Browser dual-agent pattern from InfoSeek (arXiv:2509.00375), adapted
for codebase exploration instead of web research.

ARGUMENTS may contain:
- A path to a local repository
- A path to a directory of raw agent dumps (XML/MD)
- "from-dumps" to skip exploration and go straight to synthesis
- "validate-only" to run validation on an existing report
- A GitHub URL (will clone first)

## Architecture

```
Phase 1: RECON         → Build exploration tree from repo structure
Phase 2: DEEP-DIVE    → Dispatch agents to explore tree branches
Phase 3: REFINE       → Condense raw dumps into structured intermediates
Phase 4: SYNTHESIZE   → Merge refined data into final report
Phase 5: VALIDATE     → Deterministic quality checks
Phase 6: REPORT       → Write final artifact with quality scorecard
```

The skill scripts directory is at: `~/.claude/skills/synthesizer/scripts/`

---

## Phase 1: RECON — Build Exploration Tree

### Goal
Create an exploration plan by analyzing the repo's structure, NOT by directory
layout, but by following the **import/dependency graph** from entry points.
This is the "Research Tree" analog from InfoSeek.

### Steps

1. **Detect language and manifest:**
   Run the detection script:
   ```bash
   python ~/.claude/skills/synthesizer/scripts/detect_project.py <repo_path>
   ```
   This outputs JSON with: language, manifest_file, entry_points, source_dirs.

2. **Build the exploration tree:**
   ```bash
   python ~/.claude/skills/synthesizer/scripts/build_exploration_tree.py <repo_path>
   ```
   This outputs `exploration_tree.json` — a tree of nodes where:
   - Root = entry point(s)
   - Children = imported/dependent modules
   - Each node has: path, type, imports, exported_symbols, estimated_complexity

   The script follows imports recursively, building the actual dependency graph.
   It handles: Go, Python, TypeScript/JavaScript, Rust, Java, C#.

3. **Generate dispatch plan:**
   ```bash
   python ~/.claude/skills/synthesizer/scripts/plan_dispatch.py <exploration_tree.json> [--agents N]
   ```
   This partitions the tree into N branches (default 4), optimizing for:
   - **Balanced load** — similar file counts per agent
   - **Cohesion** — related packages stay together
   - **Coverage** — every node assigned to exactly one agent

   Outputs `dispatch_plan.json` with agent assignments.

4. **Read the dispatch plan** and proceed to Phase 2.

If the exploration tree script fails or the language is unsupported, fall back
to directory-based slicing (the legacy approach). Document which approach was used.

---

## Phase 2: DEEP-DIVE — Dispatch Agents

### Goal
Send documenter agents to explore their assigned tree branches. Each agent
follows the InfoSeek-inspired `<think> → <read> → <information>` cycle.

### Agent Prompt Template

For each agent, use this prompt structure (fill in the specifics from the dispatch plan):

```
You are a codebase analyst performing an EXHAUSTIVE deep-dive of [{BRANCH_NAME}]
in the [{PROJECT_NAME}] project ([{LANGUAGE}]).

The repo is at: [{REPO_PATH}]

## YOUR ASSIGNED FILES
{FILE_LIST from dispatch_plan.json}

## METHODOLOGY — Think-Read-Refine Cycle

For EACH file you read, follow this cycle:

### Think Phase
Before reading, write a <think> block:
- WHY am I reading this file? What do I expect to find?
- What connections to previously read files should I look for?
- What questions do I need this file to answer?

### Read Phase
Read the file. Extract:
- All exported types, interfaces, structs, classes
- All exported functions/methods with signatures
- Constants and enumerations
- SQL queries (if any)
- Configuration keys (if any)
- Error types and handling patterns

### Refine Phase
After reading, write an <information> block:
- What did I learn that was expected?
- What was surprising or non-obvious?
- What new questions emerged?
- What files should I prioritize reading next?

## OUTPUT FORMAT

Write TWO files:

1. XML at: [{OUTPUT_DIR}]/raw/{NN}-{branch-name}.xml
   Using the template at: ~/.claude/skills/synthesizer/templates/agent-dump.xml

2. MD at: [{OUTPUT_DIR}]/raw/{NN}-{branch-name}.md
   Same content in readable markdown.

## EXHAUSTIVENESS REQUIREMENTS
- Document EVERY file in your assignment
- Include EVERY exported symbol
- Include actual code snippets for important patterns
- Include actual SQL queries verbatim
- Do NOT summarize — be exhaustive
- Target 30-50K tokens of output

## BORROWABLE IDEAS
If a current-project context is provided, include a <borrowable-ideas> section
analyzing what patterns from this codebase could apply.
```

### Dispatch Rules

1. Use `subagent_type: documenter` for all agents
2. Run all agents in **parallel** (single message, multiple Agent tool calls)
3. **NEVER** run agents in background
4. Each agent gets a descriptive `name` (e.g., "synth-cli-explorer")
5. Wait for ALL agents to complete before proceeding

### Think Phase Enforcement

The agent prompt includes the think-read-refine cycle. The XML template has
explicit `<think>` and `<information>` elements for each file. This matches
InfoSeek's explicit thinking phase before each search action.

## Phase 3: REFINE — Condense Raw Dumps

### Goal
Process raw agent dumps into a unified intermediate representation.
This is the "Refiner Agent" analog from InfoSeek.

### Steps

1. **Validate raw dumps exist:**
   ```bash
   python ~/.claude/skills/synthesizer/scripts/validate_dumps.py <output_dir>/raw/
   ```
   Checks: all expected files exist, XML is well-formed, minimum size thresholds met.

2. **Merge XML dumps:**
   ```bash
   python ~/.claude/skills/synthesizer/scripts/merge_xml.py <output_dir>/raw/ -o <output_dir>/merged.xml
   ```
   This:
   - Parses all XML dumps
   - Deduplicates entities (same type/function documented by multiple agents)
   - Resolves cross-references (Agent 1 mentions `Storage.CreateIssue`, Agent 2 documents its implementation)
   - Builds a unified entity index
   - Outputs merged.xml with a `<cross-references>` section

3. **Generate entity index:**
   ```bash
   python ~/.claude/skills/synthesizer/scripts/build_entity_index.py <output_dir>/merged.xml -o <output_dir>/entity_index.json
   ```
   Outputs a flat index of every documented entity:
   ```json
   {
     "types": ["Issue", "Storage", "DoltStore", ...],
     "functions": ["CreateIssue", "GetReadyWork", ...],
     "interfaces": ["Storage", "DoltStorage", "Transaction", ...],
     "constants": ["StatusOpen", "TypeBug", ...],
     "files_documented": ["cmd/bd/main.go", ...],
     "files_total": 245,
     "files_covered": 198,
     "coverage_pct": 80.8
   }
   ```

## Phase 4: SYNTHESIZE — Write Final Report

### Goal
Read the merged XML and entity index, then write the final architecture report.

### Report Structure

The final report MUST contain these sections (matching learn-repo skill output
but with deeper content from multi-agent analysis):

1. **Overview** — what the project is, problem it solves, tech stack, scale metrics
2. **Architecture** — directory layout mapped to responsibilities, layer diagram
3. **Core Abstractions** — every major type, interface, and pattern with code examples
4. **Data Flow** — end-to-end traces of key operations (create, read, update, delete)
5. **Design Patterns** — catalog of patterns used, with rationale and examples
6. **Dependency Graph** — how internal packages depend on each other
7. **Quality Scorecard** — validation results from Phase 5 (added after validation)

If project-aware mode:
8. **Borrowable Ideas** — patterns applicable to the current project, with specifics

### Synthesis Process

1. Read `merged.xml` — this is your primary source
2. Read `entity_index.json` — use for completeness checking as you write
3. For each section, pull from ALL agent reports (not just one)
4. Cross-reference data flows: trace from CLI → service → storage → SQL
5. Include actual code snippets for every pattern described
6. Include actual SQL for database-heavy projects

### Output Location

Write to: `{output_dir}/{project_name}-architecture.md`

The output_dir is typically `dev-notes/{project_name}/` or `knowledge/` depending
on user preference.

## Phase 5: VALIDATE — Deterministic Quality Checks

### Goal
Run automated validation scripts against the report and raw dumps.
This is the quality assurance analog from InfoSeek's difficulty/verifiability checks.

### Validation Suite

Run ALL validators:

```bash
python ~/.claude/skills/synthesizer/scripts/validate_coverage.py \
  --repo <repo_path> \
  --dumps <output_dir>/raw/ \
  --report <output_dir>/{project}-architecture.md \
  --output <output_dir>/validation/coverage.json

python ~/.claude/skills/synthesizer/scripts/validate_completeness.py \
  --repo <repo_path> \
  --entity-index <output_dir>/entity_index.json \
  --output <output_dir>/validation/completeness.json

python ~/.claude/skills/synthesizer/scripts/validate_consistency.py \
  --dumps <output_dir>/raw/ \
  --output <output_dir>/validation/consistency.json

python ~/.claude/skills/synthesizer/scripts/score_report.py \
  --coverage <output_dir>/validation/coverage.json \
  --completeness <output_dir>/validation/completeness.json \
  --consistency <output_dir>/validation/consistency.json \
  --output <output_dir>/validation/scorecard.json
```

### Validator Details

**validate_coverage.py** — File Coverage Check
- Lists all source files in the repo (language-aware, excludes tests/vendor/generated)
- Checks which files are mentioned in the raw dumps
- Reports: covered files, uncovered files, coverage percentage
- PASS threshold: >= 70% of source files mentioned

**validate_completeness.py** — Symbol Documentation Check
- Extracts public symbols from the repo (exported functions, types, interfaces)
- Cross-references against entity_index.json
- Reports: documented symbols, undocumented symbols, completeness percentage
- PASS threshold: >= 60% of public symbols documented

**validate_consistency.py** — Cross-Reference Consistency Check
- Extracts entity names from each XML dump
- Checks: same entity described consistently across dumps
- Detects: contradictions, missing cross-references, orphan references
- Reports: consistent references, inconsistent references, orphans
- PASS threshold: 0 contradictions

**score_report.py** — Overall Quality Score
- Aggregates all validation results
- Computes weighted score: coverage(30%) + completeness(30%) + consistency(40%)
- Grades: A (>=90), B (>=75), C (>=60), D (>=45), F (<45)
- Outputs scorecard JSON with grade, subscores, and actionable recommendations

### Handling Failures

If any validator produces warnings or failures:

1. Read the validation output to understand what was missed
2. Determine if the gap is significant (a missing utility file is fine; a missing core interface is not)
3. For significant gaps: dispatch a **targeted follow-up agent** to read the missing files
4. Re-run synthesis and validation
5. Maximum 2 remediation rounds (to prevent infinite loops)

---

## Phase 6: REPORT — Final Artifact

### Goal
Append the quality scorecard to the architecture report and clean up.

### Steps

1. Read `validation/scorecard.json`
2. Append a "## Quality Scorecard" section to the architecture report:
   ```markdown
   ## Quality Scorecard

   | Metric | Score | Details |
   |---|---|---|
   | File Coverage | 85.2% | 198/232 source files documented |
   | Symbol Completeness | 72.1% | 340/471 public symbols documented |
   | Cross-Reference Consistency | 100% | 0 contradictions across 4 agent reports |
   | **Overall Grade** | **B (82.4)** | |

   ### Uncovered Areas
   - internal/formula/loops.go — loop expression evaluator
   - internal/formula/conditions.go — condition evaluator
   ```

3. Clean up temporary files:
   - Delete the cloned repo (if it was cloned by this skill)
   - Keep: raw dumps, merged.xml, entity_index.json, validation/, final report
   - These are the permanent artifacts

4. Report the final output location to the user.

## Mode: from-dumps

If the user says "from-dumps" or raw dumps already exist:

1. Skip Phase 1 (RECON) and Phase 2 (DEEP-DIVE)
2. Start at Phase 3 (REFINE) using existing dumps
3. Continue through Phase 4-6 normally

Detect dump location:
- Check `dev-notes/*/raw/*.xml` in the current project
- Check any path the user provides
- Ask if ambiguous

---

## Mode: validate-only

If the user says "validate-only":

1. Skip Phases 1-4
2. Run Phase 5 (VALIDATE) against existing report and dumps
3. Output scorecard only

---

## Fallback Behavior

If the Python scripts fail (missing Python, permission issues, etc.):

1. Log the error clearly
2. Proceed with manual equivalents:
   - Coverage: use Glob to list source files, Grep to check mentions in dumps
   - Completeness: use Grep to extract `^func `, `^type `, `class `, `def ` from repo, cross-reference
   - Consistency: read each dump and manually check entity names
3. Note in the final report that validation was done manually

## Configuration

The skill respects these conventions from the user's CLAUDE.md:
- No copyright headers on generated files
- No emojis in source code or scripts
- Conventional commits if any git operations needed
- Delete temp files after use
- Never run agents in background
- Use best-matching subagent_type (documenter for these agents)
